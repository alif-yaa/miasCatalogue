عناوين مترجمة

It is possible to provide multiple RG numbers in order to handle duplicates in OY. The Alt. RG system may need revision. The xml:id is the normalized form of the title, which is usually the first meaningful word without any diacritical marks. In cases of ambiguity the beginning of the next word is given following a dash, e.g. hizb-wiq and hizb-ahad.

Most of Ibn al-ʿArabī's titles, in keeping with the conventions of his time, have a rhyming bipartite structure. The uniform title should generally be the first part of the title, which is usually the case in the File Maker database. The full title should consist of both rhyming elements, preferably as given by Ibn al-ʿArabī in the text of the work itself.

It is possible to provide multiple variant titles, most of which will be automatically imported from the variants found on the MSS, though others, e.g. unusual variants mentioned by Ibn al-ʿArabī in the Fihrist or Ijāza, may also be recorded here even if there is no MS witness bearing that particular title. Instances when IA gives a work an alphanumeric title should also be given here as a variant. It is also possible to give multiple translated titles (in multiple languages), as well as indicating the person responsible for that translation.

This green box is an additional field for discussion of the title of the work. Discursive content has generally been distinguished by the colouration of the box.

Currently, the File Maker database only includes an incipit (usually the first part of the ḥamdala), but it may be useful to also include an explicit, especially as the first page or pages of a text often go missing. It is also possible to encode other incipits (e.g. complete khuṭba, first line after amma baʿd).

The new database also allows for the same incipit to be encoded in both Arabic and transliteration. Connected to this point, the architecture is in place to make the database fully bilingual (or even multilingual), if this is something the Trustees would like to see in the future.

This is an optional field for further discussion of incipits, explicits, and other textual identifiers.

It is possible in the new database to give multiple authors as well as multiple "editors" or other relations to the work (see below). Roles are inputted according to the MARC Code List for Relators, including: abr (abridger), ant (bibliographic antecedent), att (attribution), cwt (commentator), com (compiler), dub (dubious attribution) etc.

Three levels of place are possible in the new database: 1. location (e.g. 'in the house of the author' or 'by the Kaʿba'); 2. settlement (name of relevant city, town or village); and 3. region (e.g. Maghreb, Anatolia). Currently, the FM database tends only to record the settlement under 'place', though for many works it will be relatively easy to integrate a more granular approach, as this information is provided in discursive fields.

In the new system dates can (and should) be entered in both Hijrī and Gregorian, and can be inputted to the level of year, month, and day in a machine-readable format, which will make querying the database for temporal data much easier. The new database also has fields for terminus post quem and terminus ante quem data, to allow for the encoding of approximate dating when the exact date is unknown. A lot of these data is provided in FileMaker though in a slightly ad hoc way, which will need to be normalized

This field is for discursive information about the composition of the work: inception (e.g. vision, companion’s question/request, etc.), original composition (date/period, place, and context first written), rewritings and revisions (later recensions, rewritings, revisions).

The Verification and Authentication fields are only currently relevant for IA's works, but they could conceivably be adapted and extended at a later date (e.g. to include works of SQ).

The Authentication field currently only contains one of three options: Holograph, Autograph, or Copy (copy from an autograph MS).

This field is for discussion about the authenticity of the work, especially useful in cases where there is no scholarly consensus or clear evidence, such as the "Probable IA" and "Unverified" categories. The usefulness of this section will become apparent once commentaries and extracts have also been migrated. I do not know of a standardised system or code list for relations between works so have devised my own, including: ext (extract), com (commentary), adp (adaptation), sum (summary), ant (antecedent). The reverse relationship is indicated by the same code with a dash: ext- (extract from), com- (commentary on), adp- (adaptation of), sum- (summary of), ant- (antecedent of).

This is an optional field for elaborating on relations to other works, which could be especially useful for extracts of works that circulate independently, or reworkings of similar material.
